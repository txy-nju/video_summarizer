以下是针对“多模态视频总结器”改进后的《初步项目构想书》。

---

## 1. 项目概览 (Project Overview)

- **核心目标**：构建一个多模态 AI 总结工具，通过同步分析视频的**语音（ASR）**与**画面（Visual Features）**，生成包含场景描述、演示步骤及核心结论的深度总结。
    
- **约束条件分析**：
    
    - **Vibe Coding 兼容性**：需要处理视频帧的提取与压缩，逻辑比纯音频复杂，需依赖成熟的 `OpenCV` 或 `FFmpeg` 库。
        
    - **性能平衡**：由于画面信息量巨大，必须在“全量抽帧”和“关键帧分析”之间做取舍，以适应本地运行的显存或 API 成本。
        

---

## 2. 需求分析 (Requirement Analysis)

- **核心功能点**：
    
    1. **智能抽帧 (Keyframe Extraction)**：自动识别画面切换，提取代表性截图。
        
    2. **视觉描述 (Image Captioning)**：利用多模态模型（如 GPT-4o, Gemini 1.5 Pro 或本地的 Qwen-VL）对图片进行文字化描述。
        
    3. **多路信息融合 (Multimodal Fusion)**：将“时间戳+语音文本”与“时间戳+画面描述”对齐，交给 LLM 统一处理。
        
- **业务逻辑简述**：
    
    视频下载 → **分支 A：音频转录** → **分支 B：关键帧提取与视觉理解** → **时序对齐** → **综合摘要生成**。
    

---

## 3. 可行性与风险评估 (Feasibility & Risks)

- **技术可行性 (Technical Feasibility)**：
    
    - **方案 A (API 派 - 推荐)**：使用 GPT-4o 或 Gemini 1.5 Pro 的原生视频理解能力。你只需把视频压小上传，它们能直接“看”视频。这对 Vibe Coding 最友好，代码量极小。
        
    - **方案 B (本地混合派)**：OpenCV 抽帧 + 本地运行小型 VLM（如 Moondream2 或 Llava-v1.5-7b）。对本地显卡有要求（建议 12G 显存以上）。
        
- **潜在风险**：
    
    - **上下文爆炸 (Context Window)**：画面描述 + 语音文本会导致 Token 数量剧增，可能超出模型单次处理上限。
        
    - **成本/延迟**：多模态 API 调用成本高于纯文本；本地推理速度可能较慢。

## 2. 功能优先级定义 (MoSCoW Matrix)

由于你是本地部署且追求 Vibe Coding 效率，我们需要严格控制“必须实现”的范围，防止需求膨胀。

### **Must-Have (必须有 - MVP 核心)**

1. **多模态 Pipeline**：能够成功下载视频，利用 OpenCV 抽帧，并将图片+音频文本（或直接视频流）发送至多模态 API。
    
2. **视觉感知总结**：生成的总结必须包含对“画面中发生了什么”的描述（例如：“画面中出现了代码编辑器”、“博主展示了产品包装”）。
    
3. **Markdown 输出**：支持层级标题、加粗、列表，确保阅读体验。
    

### **Should-Have (应该有 - 体验增强)**

1. **时间戳锚点**：在总结中插入 `[02:15]` 格式的时间戳，方便回溯。
    
2. **关键帧可视化**：在摘要下方展示抽取的图片，证明 AI 确实“看”到了这些瞬间。
    
3. **异常处理**：处理网络超时、视频过长导致的 API 报错。
    

### **Could-Have (可以有 - 长期扩展)**

1. **多语言翻译**：总结后自动翻译成中文/英文。
    
2. **局部放大识别**：对视频中的特定区域进行 OCR 识别（如屏幕右下角的小字）。
    
3. **持久化存储**：将总结结果存入本地 `.md` 文件。
    

### **Won't-Have (本次不做)**

1. 用户账号系统（完全本地运行）。
    
2. 云端部署与高并发支持。
    
3. 视频剪辑功能。
