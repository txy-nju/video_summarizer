## 1. 技术选型 (Technology Stack)

基于“本地运行”与“快速构建大模型应用”的核心诉求，我们采用全 Python 栈，最大化利用现有的 AI 生态。

- **核心框架**：
    
    - **全栈框架**：**Streamlit**。完美契合“左右双栏”的原型要求，内置丰富的交互组件（如 `st.spinner`, `st.sidebar`），极大地降低了前端编写成本，非常适合构建此类 AI 代理的 MVP。
        
- **数据存储**：
    
    - **持久层**：**本地文件系统 (OS File System)**。通过 Python内置的 `tempfile` 模块管理下载的临时视频与抽取的帧，用后即焚；针对“Could-Have”的持久化需求，直接将结果写入本地 `.md` 文件，零数据库依赖。
        
    - **状态管理**：**Streamlit `session_state`**。用于在组件刷新时保持 API Key、处理进度和生成的总结文本不丢失。
        
- **核心依赖库 (中间件替代品)**：
    
    - **视频获取**：`yt-dlp`（兼容性强，支持众多主流视频平台解析与下载）。
        
    - **媒体处理**：`OpenCV` (`cv2`) 负责高效抽帧；`ffmpeg-python` 或 `moviepy` 负责音频剥离。
        
    - **大模型交互**：`openai` SDK (兼容 GPT-4o 和第三方 API) 与 `google-generativeai` SDK (对接 Gemini 1.5 Pro)。
        

## 2. 逻辑架构设计 (Logical Architecture)

为了保证代码的模块化（便于 AI 编程助手理解和维护），系统分为三个核心层：

- **UI 表现层 (Presentation Layer)**：
    
    - 完全由 Streamlit 渲染。负责接收用户输入（API Key、URL、参数），展示处理状态指示器，以及渲染最终的 Markdown 报告和图片流。
        
- **调度层 (Coordinator Layer)**：
    
    - 负责串联各个处理步骤，管理 `st.session_state`。它像一个状态机，控制应用从“待机” -> “下载中” -> “处理中” -> “展示结果”的流转。
        
- **管道处理层 (Pipeline Layer)** _(严格对应核心业务)_：
    
    - **Downloader (下载器)**：获取视频源文件并存入临时目录。
        
    - **Extractor (提取器)**：分离出音频文件 (Audio.mp3) 和关键帧序列 (Frames [img1, img2...])。
        
    - **Transcriber (转录器)**：调用 Whisper API 将音频转化为 ASR 文本。
        
    - **Synthesizer (合成器)**：将 ASR 文本、Base64 编码的图片序列与 System Prompt 组装成多模态请求。
        

## 3. 关键业务流程 (Key Workflows)

这是本架构中**优先级最高 (Must-Have)** 的核心链路——多模态分析流水线：

1. **输入触发**：用户提交视频 URL 并点击“开始分析”。
    
2. **异步清理与下载**：清理上一轮残留的临时文件，调用 `yt-dlp` 将视频下载到本地临时文件夹。UI 呈现“正在抓取音频/视频...”。
    
3. **媒体分离 (并行或顺序执行)**：
    
    - **分支 A (听觉)**：提取音频轨道，调用 OpenAI Whisper API，获得带有时间戳的转录文本（例如 VTT 或 SRT 格式，为提取 `[02:15]` 锚点做准备）。
        
    - **分支 B (视觉)**：读取用户设置的“抽帧频率 (fps)”，利用 OpenCV 计算对应帧的位置并截取。
        
4. **数据清洗与压缩**：将截取的帧缩小分辨率（如长边限制在 768px）并转换为 Base64 编码，防止触发大模型 API 的 Payload 限制。
    
5. **多模态对齐与请求**：构建 Prompt。将“带时间戳的 ASR 文本”与“按时间顺序排列的 Base64 关键帧”打包发送至 GPT-4o / Gemini 1.5 Pro。
    
6. **流式输出 (Streaming)**：接收大模型返回的 Markdown 块，利用 Streamlit 的 `st.write_stream` 在右侧核心摘要区实现“打字机”效果。并在下方通过 `st.image` 渲染关键帧画廊。
    

## 4. 系统非功能性设计 (Non-Functional Requirements)

虽然是本地应用，但依然需要保证健壮性与安全性：

- **安全性 (API 密钥保护)**：
    
    - 使用 Streamlit 的 `st.text_input("API Key", type="password")`。变量仅存在于当前运行内存中，进程结束即销毁，绝不硬编码或写入本地日志。
        
- **扩展性 / 性能优化 (防 OOM 与防报错)**：
    
    - **Payload 瘦身**：高频抽帧极易导致图片体积过大。架构必须在 OpenCV 抽帧后加入**缩放压缩 (Resize & Compress)** 环节，保证每次传递给 API 的图片总大小不超过大模型的限制（通常建议控制在 10MB-20MB 以内）。
        
    - **异常与重试机制**：由于依赖外部 API，网络抖动是常态。在调用大模型和 Whisper 时，必须使用 `tenacity` 等库加入指数退避重试机制 (Exponential Backoff)，优雅处理超时。
